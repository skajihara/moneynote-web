## AI駆動開発 プロジェクト企画書

**作成日：** 2026年4月  
**対象：** 上司レビュー用・確定版  
**開発スタイル：** AIエージェント駆動開発（Claude Code）

---

## 1. プロジェクト概要

### 1.1 背景・目的

スマホアプリ「シンプル家計簿 MoneyNote」（株式会社コモレビ）は、シンプルな操作性と充実した分析機能で多くのユーザーに支持されている家計簿アプリである。しかし同アプリはスマートフォン専用であり、PC・タブレットのブラウザから利用できるWeb版は存在しない。

本プロジェクトでは、MoneyNoteのコンセプト（シンプル・続けやすい・分析が充実）を踏襲しつつ、**Webブラウザから使える家計簿管理アプリ**を開発する。

また本プロジェクトの主目的は、**Claude Codeを用いたAI駆動開発の実践的習得**であり、要件定義・設計・実装・レビュー・運用保守・エンハンスの全工程をAIエージェントに担わせ、人間は意思決定と承認に徹するスタイルで進める。

### 1.2 開発スタイルの定義

```
【従来の開発】
人間がコードを書く → AIに質問する（補助的）

【本プロジェクトの開発スタイル】
AIエージェントがコードを書く → 人間が承認する（ディレクター）
```

| 役割 | 担当 |
|---|---|
| 要件定義・仕様策定 | 人間（最終承認）＋ AIエージェント（案の提示） |
| 設計・アーキテクチャ | AIエージェント（アーキテクトロール） |
| 実装・テストコード生成 | AIエージェント（エンジニアロール） |
| コードレビュー | AIエージェント（レビュアーロール） |
| セキュリティチェック | AIエージェント（SREロール） |
| リファクタリング | AIエージェント（リファクタロール） |
| エンハンス計画 | AIエージェント（PМロール）＋ 人間（承認） |
| 動作確認・最終判断 | **人間のみ** |

**MoneyNote Web**（仮称）

- 家計管理をPCブラウザで行いたい個人ユーザー

### 機能① 収入・支出の入力・一覧管理

#### 何ができるか
- 収入・支出を日付・金額・カテゴリ・メモで記録する
- 記録した明細を月別に一覧表示する
- 明細を編集・削除できる
- キーワード・カテゴリ・期間でフィルタリングできる

#### 何の役に立つか
家計管理の最も基本的な操作であり、「いつ・何に・いくら使ったか」を正確に把握できるようになる。一覧で見ることで無駄な支出に気づきやすくなる。

#### なぜ必要か
家計簿アプリの根幹機能であり、全ての分析・レポート機能はここに蓄積されたデータを基にしている。この機能なしには他の機能が成立しない。

#### UI概要
```
┌─────────────────────────────────────────────┐
│  2026年4月の明細               [＋ 追加]     │
├──────┬──────────┬──────────┬────────────────┤
│ 日付 │ カテゴリ │   金額   │    メモ        │
├──────┼──────────┼──────────┼────────────────┤
│ 4/15 │ 🍔 食費  │ -¥1,200  │ ランチ         │
│ 4/14 │ 🚃 交通費│  -¥540   │ 定期外         │
│ 4/10 │ 💰 給与  │ +¥280,000│ 4月分給与      │
└──────┴──────────┴──────────┴────────────────┘
```

### 機能② 月別・年別の分析レポート

#### 何ができるか
- 月ごとの収入合計・支出合計・収支差額を表示する
- 年間の月別推移をグラフで表示する
- 前月・前年同月との比較を表示する
- 支出トレンド（増加・減少傾向）を可視化する

#### 何の役に立つか
「今月はいくら使ったのか」「先月より増えているのか」が一目でわかる。年間トレンドを見ることで、季節性の支出（夏の光熱費、年末の出費増など）を事前に把握し、計画的な家計管理ができる。

#### なぜ必要か
単なる記録帳ではなく「家計を見直すツール」として機能させるために不可欠。MoneyNoteの公式サイトでも「分析レポートが充実」を主要な価値として訴求している。

#### UI概要
```
┌──────────────────────────────────────────────┐
│  2026年 年間レポート                          │
│                                              │
│  収入合計  ¥3,360,000   支出合計  ¥2,890,000 │
│  年間収支  +¥470,000                         │
│                                              │
│  月別収支グラフ（棒グラフ）                   │
│  ████ ████ ████ ████ ████ ...               │
│  1月  2月  3月  4月  5月                     │
└──────────────────────────────────────────────┘
```

### 機能③ カテゴリ別集計・円グラフ

#### 何ができるか
- 支出をカテゴリ別（食費・交通費・光熱費など）に集計する
- 円グラフで各カテゴリの割合を視覚的に表示する
- カテゴリをユーザーが自由にカスタマイズできる
- 月別・年別のカテゴリ別推移を確認できる

#### 何の役に立つか
「何に一番お金を使っているのか」が視覚的に把握できる。円グラフで割合が一目でわかるため、「食費が支出全体の40%を占めている」といった気づきが得やすい。家計の見直しポイントを特定するのに直接役立つ。

#### なぜ必要か
金額の羅列だけでは全体像が掴みにくく、継続的な家計管理が苦痛になりやすい。視覚的なグラフは家計管理への意欲を維持する効果がある。

#### UI概要
```
┌──────────────────────────────────────────────┐
│  2026年4月 カテゴリ別支出                     │
│                                              │
│     🥧 円グラフ          カテゴリ別一覧       │
│    ╭──────────╮         🍔 食費    ¥45,000  │
│   ╱  食費 35%  ╲         🚃 交通費  ¥12,000  │
│  │  住居費 28%  │        🏠 住居費  ¥80,000  │
│   ╲  交通 10% ╱         💡 光熱費  ¥15,000  │
│    ╰──────────╯         👕 衣服費  ¥8,000   │
└──────────────────────────────────────────────┘
```

### 機能④ 固定費・定期収入の管理

#### 何ができるか
- 毎月発生する固定費（家賃・保険・サブスク等）を登録する
- 毎月の定期収入（給与・副業収入等）を登録する
- 登録した固定費・定期収入を指定日に自動で明細に反映する
- 固定費の合計を一覧で確認できる

#### 何の役に立つか
毎月同じ支出・収入を手動入力する手間が省ける。また固定費の総額を把握することで、「可処分所得（自由に使えるお金）」が明確になる。固定費の見直しにも役立つ。

#### なぜ必要か
家計管理の継続の最大の障壁は「毎回の入力の手間」である。定期的な収支を自動化することで継続率が大幅に向上する。MoneyNoteでも固定費管理は主要機能として位置づけられている。

#### UI概要
```
┌──────────────────────────────────────────────┐
│  固定費・定期収入の管理          [＋ 追加]    │
├──────────────────────────────────────────────┤
│  🏠 家賃          毎月1日   -¥80,000  [編集] │
│  📱 スマホ代      毎月25日  -¥5,500   [編集] │
│  🎬 Netflix       毎月15日  -¥1,490   [編集] │
│  💰 給与          毎月25日  +¥280,000 [編集] │
├──────────────────────────────────────────────┤
│  固定支出合計  ¥86,990 / 月                  │
└──────────────────────────────────────────────┘
```

### 機能⑤ 予算設定・予算オーバー警告

#### 何ができるか
- カテゴリごとに月の予算上限を設定する
- 現在の支出が予算に対して何%消化しているかを表示する
- 予算の80%・100%超過時にアラートを表示する
- ダッシュボードで全カテゴリの予算消化率を一覧表示する

#### 何の役に立つか
「予算を決めて、それを守る」という家計管理の基本行動を支援する。予算オーバーに事前に気づくことで、月末の使いすぎを防ぐことができる。

#### なぜ必要か
記録するだけでは家計は改善しない。「目標を設定して管理する」機能があって初めて、家計簿アプリが行動変容ツールになる。

#### UI概要
```
┌──────────────────────────────────────────────┐
│  今月の予算状況（4月）                        │
├──────────────┬────────┬────────┬────────────┤
│ カテゴリ     │ 予算   │ 消化   │ 残り       │
├──────────────┼────────┼────────┼────────────┤
│ 🍔 食費      │ ¥40,000│ ¥35,000│ ¥5,000  🟡│
│ 🚃 交通費    │ ¥15,000│ ¥8,000 │ ¥7,000  🟢│
│ 🎮 娯楽費    │ ¥10,000│ ¥11,500│ -¥1,500 🔴│
└──────────────┴────────┴────────┴────────────┘
  ⚠️ 娯楽費が予算をオーバーしています
```

### 機能⑥ CSVエクスポート・インポート

#### 何ができるか
- 明細データをCSV形式でエクスポートできる
- 他アプリからエクスポートしたCSVをインポートできる
- エクスポート時は期間・カテゴリでフィルタリングできる
- インポート時は列マッピングをAIが自動補正する

#### 何の役に立つか
Excelや他の家計簿アプリとのデータ連携ができる。確定申告や家計の振り返りのためにデータを手元に持てる安心感がある。他アプリからの乗り換えが容易になる。

#### なぜ必要か
データをシステム内に閉じ込めるアプリはユーザーから敬遠されやすい。「いつでも自分のデータを取り出せる」という透明性がユーザーの信頼につながる。また他アプリからの移行ハードルを下げる。

### 機能⑦ AIによる支出分析・アドバイス

#### 何ができるか
- 月の支出データをAIが分析して自然言語で評価を提示する
- 「先月と比べて食費が15%増加しています」などのインサイトを自動生成する
- 支出削減のための具体的なアドバイスをAIが提案する
- 年間の支出パターンから来月の支出予測を表示する

#### 何の役に立つか
数字を見るだけでは気づきにくい傾向や問題点を、AIが言語化して教えてくれる。自分では気づかなかった支出の癖や改善点を発見できる。

#### なぜ必要か
本プロジェクトの核心的なAI活用機能であり、他の一般的な家計簿アプリとの最大の差別化ポイントとなる。AI駆動開発の学習という観点でも、Claude APIを使ったLLM連携の実装経験を積む重要な機能である。

```
│    外食の頻度が増えているようです。           │

│  💡 アドバイス                               │
```

### 機能⑧ ダッシュボード（全機能の入口）

#### 何ができるか
- 今月の収入・支出・収支の概要を一目で確認できる
- 予算消化率を信号機（🟢🟡🔴）で直感的に表示する
- 最近の明細5件を表示する
- AIアドバイスのサマリーを表示する
- 各機能へのナビゲーションを提供する

#### 何の役に立つか
アプリを開いた瞬間に「今月の家計状況」が把握できる。毎日チェックする習慣がつきやすくなる。

#### なぜ必要か
家計管理の継続性を高めるためのハブ画面として必須。ユーザーが最初に目にする画面であり、アプリの第一印象を決定する。

### 機能⑨ ユーザー認証・アカウント管理

#### 何ができるか
- メールアドレス・パスワードでアカウント登録・ログインできる
- パスワードのリセットができる
- プロフィール（名前・月の開始日設定）を編集できる
- アカウントを削除できる（全データ削除）

#### 何の役に立つか
複数のユーザーがそれぞれ独立したデータで利用できる。どのPCからでも自分のデータにアクセスできる。

#### なぜ必要か
マルチアカウント対応の前提として必須。セキュリティ・セキュアなデータ管理の実装経験を積む技術学習的な観点でも重要。

## 4. UI全体設計方針

### 4.1 デザインコンセプト

- **シンプル・直感的：** 操作方法に迷わない明快なUI
- **情報の優先順位：** 最重要情報を最上部・最大サイズで表示
- **信号機モデル：** 数値を読まなくても色で状態がわかる
- **モバイルフレンドリー：** スマホブラウザでも使いやすいレスポンシブ設計

### 6.2 CLAUDE.md（エージェントへの憲法）

プロジェクトルートに配置する`CLAUDE.md`は、Claude Codeが常時参照する最重要ファイルである。以下の内容を定義する。

```markdown
# CLAUDE.md

## プロジェクト概要
MoneyNote Web - Web版家計簿管理アプリ
複数ユーザー対応のマルチアカウント構成

## 技術スタック
- Backend: Spring Boot 3.x / Java 21
- Frontend: Next.js 14 / TypeScript / Tailwind CSS
- DB: PostgreSQL 16
- Cache: Redis
- AI: Spring AI + Claude API
- Batch: Spring Batch
- CSV: Apache Commons CSV
- Test: JUnit5 + Testcontainers + MockMvc
- Infra: Docker Compose

## アーキテクチャ原則
1. レイヤードアーキテクチャ（Controller → Service → Repository）
2. ドメインロジックはServiceに集約
3. 重い処理（AI呼び出し・CSV処理）は非同期化
4. AIはモック切替可能にする（application-dev.yml）

## コーディング規約
- Java: Google Java Style Guide
- TypeScript: ESLint + Prettier準拠、any型禁止
- API: RESTful、URLは複数形・小文字・ハイフン区切り
- エラー: カスタム例外クラス必須（RuntimeException直接使用禁止）
- レスポンス形式: { data, error, timestamp }

## テストの必須ルール
- 全Serviceクラスにテスト必須
- DBを使うテストはTestcontainersを使用
- MockMvcで全APIエンドポイントのテストを作成
- テストなしの実装は行わない

## セキュリティルール
- パスワード: BCryptでハッシュ化
- JWT: アクセストークン15分・リフレッシュトークン7日
- SQLインジェクション: JPQLまたはCriteriaAPI使用
- 機密情報をログに出力しない

## エージェント行動ルール
1. 実装前に必ず設計をMarkdownで出力する
2. 不明点は実装を止めて質問リストを出力する
3. 一度に変更するファイルは5つまで
4. 実装とテストは必ずセットで作成する
5. エラー時は原因・影響・修正方針を先に説明する
```

### 6.3 Claude Code カスタムコマンド

`.claude/commands/` に定義するカスタムコマンドにより、毎回のロール宣言を省略し、一貫した指示を再利用できる。

| コマンド | 役割 | 主な用途 |
|---|---|---|
| `/design [機能名]` | アーキテクト | 設計書・ER図・シーケンス図の生成 |
| `/implement [設計書]` | バックエンドエンジニア | 設計書に基づく実装・テスト生成 |
| `/frontend [仕様]` | フロントエンドエンジニア | Next.jsコンポーネント・画面実装 |
| `/review` | シニアレビュアー | コードレビュー（Critical/Major/Minor） |
| `/security` | SRE | セキュリティ診断・OWASP Top10確認 |
| `/refactor` | リファクタエンジニア | 品質改善（機能変更なし） |
| `/test` | QAエンジニア | テストケース追加・カバレッジ向上 |
| `/ops [エラー内容]` | 運用エンジニア | エラー分析・原因特定・修正提案 |

**コマンド使用例：**
```
# 設計フェーズ
/design 収入・支出の入力・一覧管理機能

# 実装フェーズ（設計承認後）
/implement docs/design/transaction.md

# レビューフェーズ
/review

# セキュリティフェーズ
/security
```

### 6.4 Claude Code Rules（ルール設定）

`.claude/settings.json` でプロジェクト固有のルールを設定し、エージェントが常に最適な行動を取れるよう制御する。

```json
{
  "rules": [
    "実装前に必ず設計書をMarkdownで出力し、ユーザーの承認を待つ",
    "テストコードなしの実装コードは生成しない",
    "セキュリティに関わる変更は必ず理由をコメントに記載する",
    "日本語でコメント・ドキュメントを記述する",
    "エラーが発生した場合は原因・影響範囲・修正方針を説明してから修正する"
  ],
  "permissions": {
    "allow": ["Bash", "Read", "Write"],
    "deny": ["production-db-access"]
  }
}
```


## 9. 懸念点・リスクと対策

| リスク | 対策 |
|---|---|
| Claude APIのコスト超過 | `ai.mock=true` で開発中はダミー応答を使用。実APIは動作確認時のみ使用 |
| Docker for Windowsの重さ | WSL2統合を有効化。メモリ割り当てを.wslconfigで調整 |
| 2ヶ月でのStep消化漏れ | Step 1〜9をMVPとして優先。Step 10〜11はエンハンスに回せる |
| エージェントの暴走・誤実装 | Gate 1〜3の承認プロセスを必ず守る。設計承認なしに実装させない |
| テストデータの品質 | seed.sh に現実的なバリエーションのダミーデータを用意する |
